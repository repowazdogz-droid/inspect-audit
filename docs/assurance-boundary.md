# Assurance boundary

This document states precisely what `inspect-audit` does and does not establish.
Read it before relying on a verdict.

## What a verdict means

- **FAIL** — at least one implemented check found a defect that, in the tool's
  model, can make an aggregate metric wrong or a log untrustworthy (e.g. samples
  dropped from a denominator, NaN scores, duplicate samples, an unreadable log).
- **WARN** — at least one check found something that undermines trust or
  comparability but does not by itself prove the aggregate is miscomputed (e.g.
  self-grading, unpinned generation settings, abstentions).
- **PASS** — every implemented check that *ran* found no issue.
- **NOT_CHECKED** — reported per check when the data it needs is absent. It is
  never folded into PASS.

## The four things this tool cannot do

1. **It cannot certify an evaluation as valid.** `PASS` means the implemented
   checks found nothing, over the subset of failure modes they model. An
   evaluation can be wrong in ways this tool does not look at: a biased or
   contaminated dataset, a mis-specified task, an unfair prompt, a grade that is
   wrong but *parseable*, a metric that is the wrong metric for the question. A
   passing audit is the absence of found defects, not the presence of validity.

2. **It cannot see what the log does not record.** It is static analysis over the
   `.eval` file. If per-sample records were not logged, sample-level checks report
   `NOT_CHECKED`. If a grader's rubric was a default (not stored in scorer
   params), the tool cannot reconstruct it. It reads what Inspect wrote, nothing
   more.

3. **It does not judge content.** There is no LLM in the loop. It will not tell
   you whether an answer was actually correct, whether a grade was reasonable, or
   whether a jailbreak "really" succeeded. Those require judgement the tool
   deliberately does not exercise.

4. **It makes no universal claims.** Every finding is about the specific log in
   front of it. The tool asserts nothing about evaluations, models, or Inspect in
   general.

## Heuristic checks

Most checks are exact reads of recorded fields. Two rely on a heuristic and are
labelled as such in their output; neither can raise a FAIL:

- **REG001** guesses whether a model runs a reasoning regime from its name. A
  false negative (an unrecognised reasoning model) is silent; a false positive is
  a WARN, never a FAIL.
- **SCO004** keys off the exact explanation prefix Inspect writes on a grader
  parse failure (`"Grade not found in model output"`). This is reliable against
  the Inspect versions it was validated on, but is a string contract that a
  future Inspect release could change.

## Read-only and deterministic guarantees

- The tool opens logs read-only via `inspect_ai.log.read_eval_log` and never
  writes, moves, or mutates a source log. This is enforced by a test that hashes
  every fixture before and after an audit.
- Output is deterministic: no wall-clock timestamps, stable ordering. The same
  log produces byte-identical reports. This too is covered by a test.

## Scope of validation

Checks are validated by mutation testing: for each check there is a fixture log
carrying exactly that defect, and the test asserts the check fires on it and does
**not** fire on a clean log. A check that cannot demonstrably fail on its own
defect is not shipped. The fixture corpus is generated by `inspect_ai`'s own
writer, so it exercises the real schema, not a mock of it.
